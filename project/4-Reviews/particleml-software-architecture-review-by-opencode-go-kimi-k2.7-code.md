# Review Report: ParticleML Software Architecture Design

**Document under review:** `docs/engineering/particleml-architecture.md`
**Reviewer:** opencode-go/kimi-k2.7-code  
**Date:** 2026-07-14  
**Review type:** Document review — architecture correctness, integration risks, and execution readiness

---

## Summary Assessment

This architecture is a disciplined, fail-closed design for a CMS 2015 top-vs-QCD feature-availability fine-tuning study. Its strengths are the single canonical full-D dataset, deterministic A-D views derived from one source, strict leakage controls, explicit E0/E0.5 gates, and separation between CMSSW extraction and Python ML tooling.

The most consequential gaps are integration-level: the architecture assumes `omnilearned train --dataset custom` can be invoked directly, but the pinned OmniLearned README requires a prior `omnilearned dataloader` step that is missing from the design. A second critical issue is that the architecture is built around CMS 2015 while the still-authoritative `Research Plan v0.3` is written for JetClass, creating a source-of-truth conflict until the plan is amended. Several other issues concern under-specified details that will block E0 or E0.5 if left unresolved: the PID-to-category mapping, checkpoint artifact identity, training-size subsetting, and the `clip_inputs` flag.

With those resolved, the design is a sound baseline for implementation.

## Overall Verdict

| Dimension | Rating | Notes |
|---|---|---|
| Architecture correctness | Good with gaps | Data contracts and leakage controls are sound, but OmniLearned CLI integration details (dataloader, `clip_inputs`, checkpoint identity) are not fully verified. |
| Consistency with upstream documents | Needs resolution | CMS-based architecture conflicts with still-authoritative JetClass research plan; AGENTS.md/README structure is stale. |
| Risk management | Strong | Fail-closed conditions, E0/E0.5 gates, and fallback policies are clearly stated. |
| Actionability | High | Most gaps can be closed by adding definitions, a dataloader step, and a plan amendment before implementation begins. |
| Reproducibility design | Strong | Canonical dataset, deterministic views, manifest hashes, run records, and aligned predictions provide a solid reproducibility backbone. |

**Action required before implementation:** add the missing `omnilearned dataloader` integration, amend the research plan to CMS 2015, define the charge-sign-free PID mapping, specify the training-size subsetting rule, and verify or remove the `clip_inputs` flag.
