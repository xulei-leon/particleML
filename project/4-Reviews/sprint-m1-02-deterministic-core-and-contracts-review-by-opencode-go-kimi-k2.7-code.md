# Review Report: Sprint M1-02 Deterministic Core and Contracts

**Document reviewed:** `project/3-Plan/sprint-m1-02-deterministic-core-and-contracts.md`

**Review type:** Document review

**Reviewer:** opencode-go/kimi-k2.7-code

**Date:** 2026-07-17

**Scope:** Review the Sprint plan against the linked functional requirements (`FR-DATA-001`, `FR-DATA-006`, `FR-REP-001`, `FR-REP-002`), the authoritative software documentation (`docs/software/specification.md`, `docs/software/architecture.md`), the existing JSON Schemas (`schemas/*.schema.json`), the development plan (`project/3-Plan/particleml-development-plan.md` Phase 1), and the preceding Sprint M1-01 completion status.

## Executive Summary

The Sprint plan is well-structured, correctly scoped to deterministic core without drifting into extraction or model behavior, and properly identifies the Phase 1 exit criterion. However, the plan has a critical gap: it references functional requirements as implementation authority but never cross-references `docs/software/specification.md`, which contains the exact algorithms (§2.1 source manifest format, §2.2 PFN split algorithm, §2.3 stable jet identity, §6.2 CLI commands, §9 error taxonomy) that developers must implement. The FRs provide high-level acceptance criteria but not the byte-level contract needed for reproducible implementation.

A second critical issue is the implied event-level overlap detection assigned to the manifest module (§5.2 "Reject PFN/event overlap") despite the Sprint explicitly excluding dataset reading (§3 "Not included: Reading ROOT/HDF5 datasets"). Event overlap detection requires (run, lumi, event) tuples that are not present in the source manifest format (§2.1). The Sprint must clarify whether event overlap is deferred to a post-extraction semantic validator or whether the manifest/split module includes a separate event-identity input.

## Conclusion

The Sprint plan is structurally sound and correctly scoped to the deterministic core. The two critical findings both concern the boundary between this Sprint's scope (manifest/split/contracts without dataset reading) and the specification's requirements (which include event-level checks that depend on extraction): the missing specification cross-references make the implementation contract ambiguous, and the assignment of event overlap detection to the manifest module contradicts the "no dataset reading" scope. Both should be resolved before implementation begins. The remaining findings cover missing golden vectors, module organization, M1-01 dependency status, and undefined terminology that should be clarified to avoid implementation drift.

## Approval Status

**Not approved as-is.** Address the two Critical findings (specification cross-references and event overlap scope clarification) and the High finding about jet identity format scope. The Medium findings should be resolved before the Sprint is considered ready for execution.
