**Date:** 2026-07-17

**Scope:** Review the Sprint plan against the linked functional requirements (`FR-DATA-005`, `FR-MODEL-003`, `FR-REP-002`), the approved development plan (`project/3-Plan/particleml-development-plan.md`), and the authoritative software documentation suite (`docs/software/architecture.md`, `docs/software/specification.md`, `docs/software/traceability-matrix.md`, `docs/software/requirements.md`). The Research Plan v0.4 (`docs/research/research-plan.md`) was also consulted for scientific context.

## Executive Summary

The Sprint plan is well-scoped, correctly limits itself to contract correction and cloud bootstrap, and preserves the local-editing/cloud-verification boundary required by the development plan. However, the plan cannot be considered internally consistent until two authoritative documents that it commits to correct still contain the obsolete one-hot PID contract. The most important blocker is `docs/software/specification.md` §4.3, which explicitly describes "View dimensions after one-hot PID expansion" and assigns Config C 11 fields (B + six PID one-hot channels). This directly contradicts the native integer-PID contract that is the central goal of the Sprint. `docs/software/architecture.md` §5.2 also contains a related one-hot conversion statement that must be reconciled.

Additional actionable findings cover field-order ambiguity in the C view, the need to strengthen `scripts/validate_software_docs.py` to detect obsolete PID language, the missing VitePress configuration that will block the planned `pnpm docs:build` check, and the need for more concrete RunPod deliverable definitions.

## Conclusion

The Sprint plan is coherent and correctly scoped, but it currently describes a contract-correction goal that is not yet achieved in the authoritative documents it commits to change. The highest-priority action before executing the Sprint is to reconcile `docs/software/specification.md` §4.3 and `docs/software/architecture.md` §5.2 with the native integer-PID contract in `FR-DATA-005`, `FR-MODEL-003`, and the development plan, and to add a validator check that prevents the obsolete one-hot language from re-entering the documentation suite. The VitePress configuration gap and the RunPod file-structure ambiguity should also be resolved so that the first cloud CI run can pass cleanly.

**Not approved as-is.** The Sprint plan should be updated to address the Critical and High findings, after which it can proceed to implementation.
