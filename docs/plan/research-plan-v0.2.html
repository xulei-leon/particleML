Research Plan v0.2

Input Feature Representations in Jet Foundation Models

A 12-week empirical study of how particle-level feature choices affect top-tagging performance, data efficiency, and transfer-learning value in PET-based jet foundation models.

01

Executive Summary

The project tests a focused claim: for jet foundation models, downstream classification performance may depend as much on the particle-level input representation as on the backbone architecture. Rather than attempting a broad foundation-model benchmark, version 0.2 narrows the work to a systematic, statistically disciplined feature-ablation study on JetClass top tagging.

Scientific question

Which particle features are necessary and sufficient for strong top-tagging performance after transfer learning from a pretrained jet foundation model?

Research product

A first-author preprint with reproducible ablation curves, paired statistical tests, and a public experiment log suitable for graduate research applications.

Scope discipline

The minimum publishable result is the OmniLearn feature-ablation grid. Baselines and multi-class tests are treated as staged controls, not prerequisites for a coherent paper.

Central contribution. The study will separate three effects that are often entangled in jet-machine-learning comparisons: input feature richness, labeled-data scale, and transfer from a pretrained backbone. A null result is still valuable: if richer representations do not materially improve AUC under controlled fine-tuning, that would constrain future claims about feature engineering in jet foundation models.

02

Research Question and Positioning

Main Research Question

How do particle-level input feature representations affect the top-tagging performance and data efficiency of a pretrained PET-based jet foundation model?

ID Question Status
RQ1 What AUC gain is obtained by adding particle ID, charge, and energy/momentum fraction features to a kinematic-only particle representation? Core
RQ2 Does richer input information improve data efficiency, measured by the fine-tuning sample size required to reach a target AUC? Core
RQ3 How much of the observed performance comes from pretrained transfer rather than from the PET architecture trained from scratch? Control
RQ4 Do the feature-ranking trends persist for one additional task, such as five-class JetClass classification? Optional

Why This Is a Good Undergraduate Research Project

The project is ambitious but bounded. It builds directly on an existing reference implementation, avoids the cost of pretraining a foundation model from scratch, and transforms the researcher's existing strengths in Python, uncertainty quantification, and Bayesian-style statistical thinking into a deep-learning project with a concrete publication path.

Interpretation boundary. Feature ablations are not causal interventions on the underlying physics. They measure performance associations under a fixed model, dataset, optimizer, and preprocessing pipeline. The paper should describe feature effects as empirical performance differences, not as proof that any single feature independently causes a change in classifier behavior.

03

Experimental Design

Dataset and Task

The primary dataset is JetClass, a large simulated jet dataset released for deep-learning studies of jet tagging. The main task is binary classification between hadronic top jets and QCD jets. This task is the best first target because it is widely recognized, computationally manageable, and directly connected to the reference literature on constituent-based jet taggers.

Component Plan
Input object Particle constituents represented as per-particle feature vectors, padded or masked to a fixed maximum number of constituents.
Primary labels Hadronic top decay signal versus QCD background.
Split policy One fixed train/validation/test manifest generated before training. All models and feature configurations use the same held-out test set.
Leakage control Store split hashes, source files, preprocessing version, and event counts in a versioned manifest.

Feature Configurations

Config Particle-Level Features Purpose
A: Kinematics Relative eta, relative phi, log transverse momentum, and mask. Minimal representation.
B: + PID Config A plus compact particle-identification encoding. Tests particle identity.
C: + Charge Config B plus electric charge. Tests signed particle information.
D: Full Config C plus energy and momentum fraction features aligned with the OmniLearn-style richer input. Reference upper bound.

Model Conditions

Condition Role Priority
Pretrained OmniLearn/PET backbone, fine-tuned end to end Main experimental system for RQ1 and RQ2. Required
Same architecture, random initialization Transfer-learning control for RQ3. Recommended
Deep Sets baseline Simple permutation-invariant baseline to check whether gains require a transformer-style backbone. Recommended
ParticleNet or EdgeConv-style baseline Stronger task-specific point-cloud comparison if implementation time allows. Optional

Run Matrix

ID Experiment Configs Data Sizes Seeds Approx. Runs
E1 Core pretrained feature ablation A, B, C, D 10^2, 10^3, 10^4, 10^5 per class 5 80
E2 Random-initialization control A, D 10^4 and 10^5 per class 3 12
E3 Deep Sets baseline A, D 10^4 and 10^5 per class 3 12
E4 ParticleNet or EdgeConv baseline A, D 10^5 per class 3 6
E5 Five-class JetClass extension A, D 10^5 total balanced sample 3 6

Minimum viable paper. E1 alone can support the central paper if executed cleanly. E2 and E3 strengthen interpretation. E4 and E5 should be attempted only after the Week 8 gate confirms that the core ablation is complete, analyzed, and producing stable figures.

04

Metrics and Statistical Analysis

Primary Metrics

Metric Use Reporting
ROC AUC Main top-tagging performance measure. Median over seeds, with bootstrap 95% confidence interval on the fixed test set.
Background rejection at fixed signal efficiency Physics-facing classifier quality measure. Report at 30% and 50% signal efficiency when stable.
Data-efficiency threshold Sample size required to reach a predeclared AUC target. Interpolate AUC versus log sample size; avoid over-interpreting extrapolation.

Comparison Strategy

  • Use paired comparisons wherever possible because all models share the same test examples.
  • Use DeLong-style correlated ROC comparison for AUC when implementation is available and validated.
  • Report absolute effect sizes, such as Delta AUC and percent change in background rejection, alongside p-values.
  • Apply a false-discovery-rate correction across the planned adjacent feature comparisons.
  • Predeclare the feature-comparison order as A to B, B to C, and C to D to avoid post hoc cherry-picking.

Figures Required for the Paper

Figure Message Data Source
Figure 1 Feature-ablation AUC across data sizes. E1
Figure 2 Delta AUC by added feature category, with uncertainty. E1 paired comparisons
Figure 3 Data-efficiency curves showing whether richer features shift the learning curve. E1
Figure 4 Transfer and baseline controls. E2, E3, optional E4

Evidence standard. The paper should avoid treating a statistically significant Delta AUC as automatically physics-significant. A small AUC change should be interpreted through background rejection, stability across seeds, and whether it persists at larger data scales.

Decision Gates

Gate Pass Criteria Action if Failed
Week 2: Reproduction Gate Reference model runs; preprocessing produces validated A-D tensors; split manifest is fixed. Freeze scope to E1 only and simplify baselines until the reference workflow is stable.
Week 5: Signal Gate At least two data scales complete; metrics are stable across seeds; no data leakage detected. Reduce seeds for small-data conditions and prioritize S3/S4, where publication-quality curves are most important.
Week 8: MVP Gate E1 complete and analyzed; E2 or E3 at least partially complete; figure drafts exist. Stop new experiments and write the paper around the core ablation plus available controls.
Week 10: Writing Gate All final figures and statistical tables are frozen. Move unresolved experiments to future work rather than extending the schedule.

Compute and Storage

Training should use rented single-GPU instances with 16-24 GB VRAM. Debugging and preprocessing can be performed locally or on low-cost CPU instances. All training runs should write a compact JSON record containing the run ID, git commit, dataset manifest hash, feature configuration, seed, hyperparameters, best validation epoch, and final test metrics.

Resource Plan
GPU RTX 3090, RTX 4090, A4000, or similar single-GPU instance.

Storage Use persistent storage for JetClass subsets, generated manifests, model checkpoints, and metric logs.
Reproducibility Version every preprocessing script and store immutable split manifests before the first reported run.
06

Risk Register

Risk Impact Mitigation
Reference implementation or checkpoint is difficult to reproduce. High Make reproduction the Week 2 gate. If unresolved, use the PET architecture with available weights only when validated; otherwise recast the paper as a controlled architecture study.
Feature encodings differ subtly across models. High Build one preprocessing library that emits all feature configurations from the same source tensors and unit-test dimensions, masks, normalizations, and class balance.
Baseline implementation consumes too much time. Medium Keep Deep Sets as the required simple baseline. Treat ParticleNet or EdgeConv as an optional extension after E1 is complete.
Observed feature effects are small. Medium Report confidence bounds and practical effect sizes. A precise null or near-null result still answers the research question.
Cloud budget or time runs out. High Use the decision gates aggressively. E1 is the only non-negotiable experiment; optional runs stop before they threaten manuscript completion.

Advisor Evidence

A concrete body of work that can support recommendation letters: reproducible experiments, weekly milestones, and a manuscript-ready result.

Manuscript Skeleton

Section Core Message
Introduction Jet foundation models promise reusable representations, but input representation choices remain under-controlled.
Methods Fixed dataset, fixed split, fixed backbone, four feature configurations, staged controls.
Results Feature-ablation curves, data-efficiency curves, transfer controls, and baseline comparisons.
Discussion What the results imply for jet tagger design, detector-feature engineering, and future foundation-model benchmarks.
Limitations Single dataset, one primary task, no causal attribution, limited compute, and simulation-only validation.
